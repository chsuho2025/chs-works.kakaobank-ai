# 멜론 AI DJ가 곡명을 정확히 발음하는 법: TTS 발음사전

## 영어와 숫자가 포함된 곡명·아티스트명 약 3만 5천 개를 발음사전 후보로 정리하고, 초기 약 10% 분량의 단어들을 수동으로 조사했습니다. 조사 기록에서 패턴을 찾고 처리 목표를 조정한 뒤, 주 단위 등록과 적용 확인까지 이어간 과정을 설명합니다.

---

## 프로젝트 소개: 음악을 읽기 위한 발음사전

TTS(Text-to-Speech)는 텍스트를 음성으로 바꾸어 읽어주는 기술입니다. 뉴스 기사처럼 일반 문장을 읽을 때도 쓰이지만, 음악 서비스에서는 추천 설명 안에 곡명과 아티스트명이 계속 등장합니다. 화면에서는 정확하게 적힌 이름도 TTS가 읽는 순간 사용자가 알아듣기 어려운 이름으로 바뀔 수 있습니다.

그 이유는 음악 이름이 일반적인 한국어 읽기 규칙만으로 처리하기 어려운 표기를 많이 포함하기 때문입니다. 여러 언어의 이름이 함께 있고, 숫자와 기호가 철자 일부로 쓰이기도 하며, 팬이나 국내 음악 시장에서 통용되는 발음이 원어의 기계적인 읽기와 다를 수도 있습니다. 클래식 곡명에는 작품의 종류, 번호, 조성과 작품 목록을 나타내는 약어가 한 문장 안에 이어집니다.

발음사전은 이런 표기를 TTS가 어떤 소리로 읽어야 하는지 별도로 등록해두는 자료입니다. 예를 들어 화면의 영문 표기는 그대로 유지하되, 음성 출력에서는 국내 사용자가 익숙하게 알아들을 수 있는 발음을 연결하는 방식입니다. 발음사전에 항목을 등록했다고 해서 작업이 끝나는 것은 아닙니다. 실제 문장에 넣어 재생했을 때 등록한 발음이 적용되는지, 앞뒤 조사와 자연스럽게 이어지는지, 의미 단위에 맞게 쉬는지 다시 들어봐야 합니다.

제가 참여한 업무는 사내 TTS에 사용할 음악 도메인 발음사전을 정리하고 적용 결과를 검수하는 일이었습니다. 여기서 `음악 도메인`은 음악 서비스에서 반복적으로 등장하는 곡명, 아티스트명, 클래식 작품 표기처럼 일반 문장과 다른 읽기 기준이 필요한 영역을 뜻합니다. 발음사전의 일부는 사내 TTS뿐 아니라 대화형 음악 추천 기능의 추천 문장을 읽는 TTS에도 적용됐습니다.

대화형 음악 추천에서는 사용자의 요청에 따라 추천 문장이 매번 달라질 수 있습니다. 그 문장 안에 어떤 곡과 아티스트가 들어갈지 미리 한정하기 어려웠기 때문에, 몇 개의 대표 이름만 고치는 방식으로는 부족했습니다. 음악 서비스에 이미 등록된 이름과 매주 새로 들어오는 이름을 함께 살피며, 반복해서 나타나는 오독을 줄일 수 있는 사전이 필요했습니다.

제가 담당한 작업 흐름을 간단히 정리하면 다음과 같습니다.

```text
영어·숫자가 포함된 곡명과 아티스트명 추출
  ↓
공개 자료를 검색해 통용 발음 수동 조사
  ↓
발음 후보와 조사 근거·소요 시간 기록
  ↓
발음사전 등록용 목록 정리
  ↓
TTS 업데이트 후 문장 단위로 재생
  ↓
Pass/Fail 확인과 재검수
```

여기서 Pass는 등록한 발음이 테스트 문장에 기대한 방식으로 적용된 경우, Fail은 등록 내용이 반영되지 않았거나 명백한 오독이 남은 경우를 뜻합니다. 발음을 조사하는 단계의 정답 표시가 아니라, 업데이트 뒤 실제 음성을 확인하는 판정이었습니다.

저는 2026년 3월부터 6월까지 카카오엔터테인먼트 인턴으로 이 작업에 참여했습니다. TTS 모델을 개발하거나 최종 읽기 규칙을 확정한 것은 아닙니다. 항목별 발음을 수동으로 조사하고, 어떤 경로에서 확인했는지와 소요 시간, 판단 근거를 기록했으며, 그 기록에서 반복 패턴을 찾아 규칙 후보에 대한 의견을 담당 멘토와 공유했습니다.

제가 담당한 범위는 다음과 같습니다.

| 구분 | 참여 내용 |
|---|---|
| 대상 추출 | 음악 서비스의 곡명·아티스트명 중 숫자와 영어가 포함된 항목 추출 |
| 수동 조사 | 검색과 공개 자료를 활용해 항목별 적정 발음 확인 |
| 조사 기록 | 검색 경로, 확인 자료, 소요 시간, 판단 근거 기록 |
| 문장 제작 | 발음사전에 등록한 단어마다 최소 한 개 이상의 테스트 문장 구성 |
| 등록 확인 | 주 1회 TTS 업데이트 후 테스트 문장으로 Pass/Fail 확인 |
| 신규 항목 | 매주 음악 서비스에 새로 등록되는 곡·아티스트 발음 추가 |
| 기록 분석 | 수동 작업 결과에서 국적·장르·표기별 반복 패턴 확인 |
| 의견 공유 | 기록 분석 결과와 규칙 후보를 담당 멘토에게 공유 |

최종 규칙 정리와 서비스 적용 확인은 담당 멘토가 주로 진행했습니다. 제가 남긴 수동 조사 기록은 이후 반복 패턴을 분석하는 데이터로 사용됐습니다.

## 표기는 맞지만 발음은 틀린 이름

프로젝트를 이해한 뒤 실제 음성에서 나타난 문제를 살펴보겠습니다. 음악 서비스의 화면에는 곡명과 아티스트명이 정확하게 표시돼 있었습니다. 그러나 같은 이름을 TTS가 읽으면 영어와 숫자를 글자 그대로 읽거나, 국내에서 통용되는 이름과 다른 발음으로 읽는 경우가 있었습니다.

텍스트로 음악을 찾을 때는 표기가 맞는지가 중요합니다. 음성으로 음악을 소개할 때는 그 표기를 사용자가 알아들을 수 있는 방식으로 전달하는지도 확인해야 했습니다. 한 글자도 틀리지 않은 곡명이 음성에서는 낯선 이름처럼 들릴 수 있었습니다.

예를 들어 클래식 곡명에는 작품명, 조성, 곡 번호와 `Op.` 표기가 한 줄에 함께 적힙니다. `Op.`는 작품 번호를 나타낼 때 널리 쓰이는 표기입니다.

> `Symphony No. 5 in C Minor, Op. 67`

사람은 표기 구조를 보고 교향곡 번호와 조성, 작품 번호를 나눠 읽을 수 있습니다. TTS가 영문 약어와 숫자를 같은 방식으로 이어 읽으면 각 숫자가 무엇을 뜻하는지 알아듣기 어렵습니다. 발음이 맞더라도 끊어야 할 위치가 어긋나면 곡명 전체가 하나의 긴 문자열처럼 들릴 수 있습니다.

<small>※ 위 곡명은 작업 방식을 설명하기 위한 공개용 대체 예시이며, 실제 사전 원문이나 내부 목록을 옮긴 것이 아닙니다.</small>

처음에는 발음이 틀린 단어를 찾아 올바른 발음을 적는 작업처럼 보였습니다. 실제로 진행해 보니 단어 하나의 발음을 고치는 것만으로 끝나지 않았습니다. 같은 단어도 문장 안에서 조사와 붙거나 숫자 앞뒤에서 호흡이 달라지면 다시 어색하게 들릴 수 있었습니다. 먼저 무엇을 조사 대상으로 삼고, 어느 수준까지 확인할지 범위를 정해야 했습니다.

## 검수 대상 설정

음악 서비스에는 매우 다양한 언어와 표기 방식이 함께 존재합니다. 한글 곡명 안에 영어가 들어갈 수 있고, 아티스트명에 숫자가 포함될 수 있으며, 클래식 곡명에는 로마자와 Op. 표기가 반복됩니다. 모든 곡명과 아티스트명을 같은 우선순위로 검수하기보다, TTS가 읽는 방식을 별도로 정해야 할 가능성이 높은 항목부터 확인했습니다.

음악 서비스에 등록된 곡명과 아티스트명 가운데 숫자와 영어가 포함된 항목을 추출했고, 그 규모는 약 3만 5천 개였습니다. 숫자와 영어가 포함됐다는 공통 조건은 있었지만 실제 문제는 여러 유형으로 나뉘었습니다.

- 영어 아티스트명을 국내 통용 발음과 다르게 읽는 경우
- 숫자를 곡명의 일부가 아니라 일반 수량처럼 읽는 경우
- 알파벳 약어를 한 단어처럼 읽는 경우
- 연도, 곡 번호, Op. 표기를 같은 방식으로 읽는 경우
- 외국어와 숫자가 이어지면서 호흡이 어색해지는 경우
- 클래식 곡명의 조성, 악장, Op. 표기를 구분하지 못하는 경우

<small>※ 약 3만 5천 개는 작업 규모를 설명하기 위한 근삿값입니다. 실제 항목 목록·내부 분류표·원본 데이터는 포함하지 않았습니다.</small>

대상이 많다고 해서 모든 항목을 같은 방식으로 처리할 수는 없었습니다. 표기가 비슷해도 맥락에 따라 읽는 법이 달라질 수 있었고, 국내에서 널리 사용하는 발음과 원어 발음 사이에서 서비스가 사용할 기준을 정해야 하는 경우도 있었습니다. 같은 표기라도 사용자는 곡명만 따로 듣지 않고 추천 이유와 안내 문장이 이어진 음성으로 듣습니다. 그래서 표기 유형만 분류하는 데서 끝내지 않고, 실제 안내 문장 안에서 이름을 알아들을 수 있는지 확인했습니다.

## 네 단계로 바뀐 작업 방식

원래 목표는 약 3만 5천 개의 항목을 빠르게 발음사전에 등록하는 것이었습니다. 처음부터 수동 작업만으로 완주하려 한 것은 아니었습니다. 사람이 발음을 찾는 과정을 세밀하게 기록하고, 그 데이터를 분석해 자동 분류에 사용할 규칙을 찾는 것이 초기 수동 작업의 목적이었습니다.

작업 방식은 크게 네 단계로 변화했습니다.

```text
수동 조사와 기록 축적
  ↓
수동 기록 분석과 패턴 발견
  ↓
AI 기반 자동 분류 시도와 기준 보완
  ↓
발음사전 등록 목적 재설정
```

이 변화는 처음 세운 계획대로 모든 항목을 처리한 과정이라기보다, 실제 작업량과 음악 고유명사의 특성을 확인하면서 방법을 수정한 과정에 가까웠습니다. 수동 작업을 충분히 해보지 않고는 자동화 기준을 만들기 어려웠고, 자동화의 한계를 확인하지 않고는 현실적인 등록 목표를 정하기 어려웠습니다.

### 1단계. 수동 조사와 기록 축적

초기 등록 방식은 100% 수동이었습니다. 각 곡명과 아티스트명에 적합한 한국어 발음을 사람이 하나씩 찾았습니다. 자동 분류 결과를 검토하는 방식이 아니라, 검색과 공개 자료를 활용해 항목별 발음을 직접 조사했습니다.

영문 표기만 보고 발음을 정하기 어려운 경우가 많았습니다. 같은 철자라도 어느 나라 언어를 기준으로 읽는지에 따라 발음이 달라질 수 있었고, 아티스트가 의도적으로 일반적인 읽기와 다른 이름을 사용하기도 했습니다. 숫자가 포함된 이름도 수량이나 연도가 아니라 알파벳을 대신하는 기호, 발음의 일부, 장르 안에서 통용되는 표현일 수 있었습니다.

이 유형은 공개된 아티스트명만 보아도 확인할 수 있습니다. `6LACK`은 숫자 6을 그대로 읽는 이름이 아니라 `black`으로 발음하고, `deadmau5`의 5는 `s`를 대신해 `dead mouse`로 읽습니다. `3OH!3`은 숫자와 문장부호가 섞여 있지만 `three oh three`, `Tech N9ne`의 9는 `nine`으로 읽습니다. 초기 활동명 `Ke$ha`의 달러 기호 역시 발음을 바꾸는 문자가 아니었습니다. 표기만 기계적으로 읽으면 아티스트가 사용하는 이름과 다른 발음이 나올 수 있는 사례입니다.

<small>※ 위 이름은 실제 내부 작업 목록이 아니라, 숫자와 기호를 포함한 고유명사의 특성을 설명하기 위해 공개된 아티스트명으로 다시 구성한 예시입니다.</small>

발음과 함께 조사 과정도 남겼습니다. 어떤 검색어를 사용했는지, 어느 자료에서 발음을 확인했는지, 한 항목을 처리하는 데 얼마나 걸렸는지, 최종 후보를 선택한 이유가 무엇인지 기록했습니다. 이 기록은 단순한 업무 일지가 아니라, 이후 자동 분류 규칙을 찾기 위한 분석 데이터였습니다.

> **수동 조사 — 공개용 재구성 예시**
>
> 표기 확인 → 아티스트 국적과 활동 정보 확인 → 국내 통용 표기 검색 → 확인 경로와 소요 시간 기록 → 발음 후보 작성

약 3만 5천 개 가운데 약 10%를 이 방식으로 진행했을 때 한계가 분명해졌습니다. 약 3,500개를 처리하는 동안 규칙 분석에 사용할 기록은 쌓였지만, 같은 방식으로 나머지 항목을 모두 조사하면 정해진 기간 안에 발음사전 등록을 마치기 어려웠습니다. 그렇다면 사람이 하나씩 확인하며 남긴 기록에서 다음 항목의 조사 시간을 줄일 단서를 찾을 수 있을까요? 이 질문을 바탕으로 수동 기록을 다시 분석했습니다.

### 2단계. 수동 기록에서 패턴 찾기

수동 검수 기록이 쌓이면서 개별 항목 사이에서 반복되는 단서를 찾을 수 있었습니다. 처음에는 서로 다른 고유명사로 보였지만, 아티스트의 국적과 음악 장르에 따라 발음을 판단할 때 참고할 수 있는 패턴이 나타났습니다.

예를 들어 알파벳 표기를 어느 나라 언어의 발음으로 읽을지는 아티스트의 국적을 하나의 판단 근거로 삼을 수 있었습니다. 같은 철자라도 영어권 아티스트와 다른 언어권 아티스트에게 동일한 읽기 기준을 적용하면 실제 이름과 멀어질 수 있었기 때문입니다.

장르도 단서가 됐습니다. 특히 힙합에서는 숫자를 알파벳이나 특정 소리를 대신하는 방식으로 사용하는 이름이 자주 나타났습니다. 숫자를 일반적인 수량으로 읽기보다 이름의 구성 방식과 장르 안의 용례를 함께 확인해야 했습니다.

다만 국적과 장르만으로 발음을 확정하지는 않았습니다. 국적이 같아도 제작자가 의도한 이름이 다를 수 있고, 다른 장르에서도 유사한 표기가 나타날 수 있었습니다. 국적과 장르는 정답을 자동으로 결정하는 기준이 아니라 수동 조사 범위를 좁히는 단서로 활용했습니다.

> **패턴 분석 — 공개용 대체 예시**
>
> - 알파벳 표기 + 아티스트 국적 → 우선 확인할 언어권 후보 설정  
> - 숫자 혼합 이름 + 음악 장르 → 숫자의 일반 읽기와 대체 표기 가능성 비교  
> - Op.가 포함된 클래식 곡명 → 반복되는 표기 구조에 따라 읽기 단위 분리  
> - 국내 통용명이 있는 고유명사 → 원어 발음과 서비스 이용자가 익숙한 발음 비교

이 단계에서 중요한 것은 사람이 만든 기록을 단순 결과물로만 두지 않은 점이었습니다. 수동 기록을 다시 분석해 다음 항목을 처리할 때 사용할 수 있는 기준 후보로 바꾸었습니다. 이렇게 찾은 패턴이 실제 처리량을 늘릴 수 있는지는 자동 분류 가능성을 검토하며 확인했습니다.

### 3단계. 자동 분류 시도와 한계

제가 남긴 수동 조사 기록에서 반복 패턴을 확인한 뒤에는 AI를 활용해 규칙에 따라 발음 후보를 자동으로 분류하는 방안을 검토했습니다. 여기서 규칙 기반 분류는 `특정 표기 구조와 조건이 나타나면 정해둔 발음 후보로 보낸다`는 기준을 적용하는 방식입니다. 약 3만 5천 개를 계속 수동으로만 처리하기 어려웠기 때문에, 국적·장르·표기 구조와 같은 단서를 활용해 사람이 확인할 범위를 줄이려는 접근이었습니다. 저는 기록 분석을 바탕으로 규칙 후보에 의견을 냈고, 최종 규칙 정리와 적용은 담당 멘토가 중심이 되어 진행했습니다.

자동 분류가 비교적 잘 맞을 수 있는 영역도 있었습니다. 표기 형식이 일정한 클래식 Op. 표기처럼 입력 구조와 읽는 단위가 반복되는 경우입니다. 반면 아티스트명과 곡명은 표기만으로 제작자의 의도를 확인하기 어려웠습니다.

문화예술 영역의 이름은 일반적인 언어 규칙을 일부러 벗어나기도 합니다. 숫자를 문자처럼 사용하거나, 철자를 변형하거나, 특정 장르와 커뮤니티 안에서만 통용되는 발음을 선택할 수 있습니다. 같은 표기라도 아티스트가 이름에 부여한 의미와 활동 맥락에 따라 읽는 법이 달라졌습니다.

이 때문에 자동 분류 결과를 그대로 발음사전에 등록하기 어려웠습니다. AI가 패턴에 따라 후보를 좁힐 수는 있었지만, 제작자의 의도와 실제 통용 발음을 확인하려면 사람의 검색과 판단이 계속 필요했습니다. 자동화는 수동 작업을 완전히 대체하기보다 사람이 확인할 우선순위를 정하는 보조 방식에 가까웠습니다.

이 시점에서 처음 세운 목표를 그대로 유지하면 정해진 시간 안에 발음사전을 모두 갱신하기 어렵다는 점이 분명해졌습니다. 처리량을 늘리기 위해 확인 절차를 생략하면 잘못된 발음을 대량으로 등록할 수 있었고, 모든 항목에서 완전한 발음을 찾으려 하면 적용 가능한 항목 수가 지나치게 줄어들 수 있었습니다. 결국 자동화 여부보다 먼저, 제한된 시간 안에 어떤 품질을 우선 확보할지 다시 정해야 했습니다.

### 4단계. 등록 목적 재설정

처음에는 각 이름의 정확한 발음을 하나씩 찾아 발음사전에 등록하는 것을 목표로 삼았습니다. 수동 조사와 자동 분류 시도를 거치면서, 정해진 시간 안에 약 3만 5천 개 전체의 제작 의도와 용례를 완전히 확인하기는 어렵다는 한계를 확인했습니다.

이 한계를 확인한 뒤에는 모든 항목의 제작 의도를 완전히 확인할 때까지 적용을 미루기보다, 확인 가능한 정확한 발음은 유지하고 사용자가 명백한 오독으로 느낄 가능성이 큰 항목을 먼저 줄이는 방향으로 우선순위를 조정했습니다.

이는 정확성을 포기한다는 뜻은 아니었습니다. 정확한 발음을 확인할 수 있는 항목은 그대로 반영하되, 확인에 많은 시간이 필요한 항목까지 모두 멈춰 두지 않도록 품질 기준의 우선순위를 다시 정한 것입니다.

글에서는 당시 판단을 다음과 같이 정리할 수 있습니다.

| 구분 | 판단 방향 | 처리 방식 |
|---|---|---|
| 정확한 발음 확인 가능 | 공식·통용 발음을 확인할 수 있음 | 확인된 발음 등록 |
| 명백한 오독 방지 가능 | 완전한 의도 확인은 어렵지만 부자연스러운 읽기는 피할 수 있음 | 우선 적용 후 추가 확인 |
| 제작 의도 확인 필요 | 밈, 조어, 예외적 표기로 자동 판단 위험이 큼 | 수동 검토 유지 |

<small>※ 위 표는 내부 등급이나 정책이 아니라, 작업 중 바뀐 판단 방향을 설명하기 위해 재구성한 자료입니다.</small>

목표를 조정한 뒤에는 반복 표기에 적용할 내부 읽기 기준을 보완했습니다. 클래식 곡명의 Op.와 악장 표기, 숫자를 알파벳이나 소리의 대체로 사용하는 경우처럼 일정한 패턴이 나타나는 항목을 유형화했습니다. 다만 구체적인 내부 기준과 예외 목록은 외부에 공개할 수 없어 이 글에서는 판단 과정과 공개 가능한 수준의 예시만 다룹니다.

이렇게 작업 목표를 조정하고 나니 다음 확인 과제도 분명해졌습니다. 사전에 발음을 등록했다는 사실만으로는 충분하지 않았습니다. 사용자가 실제로 듣는 문장 안에서도 그 발음이 자연스럽게 적용되는지 확인해야 했습니다.

## 단어가 아닌 문장으로 검수한 이유

발음사전은 단어를 등록하는 작업이지만, 사용자는 단어만 따로 듣지 않습니다. TTS는 “다음 곡은 ○○입니다” 또는 “○○의 음악을 추천해드릴게요”처럼 완성된 문장 안에서 곡명과 아티스트명을 읽습니다.

단어를 단독으로 들었을 때는 자연스러워도 문장에서는 문제가 나타날 수 있었습니다.

- 앞말과 곡명 사이에서 쉬지 않아 이름의 시작이 불분명해짐
- 곡명 뒤의 조사와 붙으면서 마지막 음절이 어색해짐
- 영어와 숫자 사이에서 불필요하게 긴 호흡이 생김
- 작품명과 Op. 표기를 구분하지 않고 이어 읽음
- 문장이 길어지면서 중간의 고유명사에서 호흡이 끊김

그래서 발음사전에 등록한 단어마다 최소 한 개 이상의 테스트 문장을 만들었습니다. 단어의 발음만 확인하는 것이 아니라, 한국어 안내 문장 안에 들어갔을 때의 발음과 호흡을 함께 듣기 위한 방식이었습니다.

> **문장 제작 — 공개용 대체 예시 1**
>
> 검수 단어: `Symphony No. 5`  
> 테스트 문장: `다음 곡은 베토벤의 Symphony No. 5입니다.`  
> 확인 내용: 영문 작품명, 숫자 읽기, 조사 앞 호흡

> **문장 제작 — 공개용 대체 예시 2**
>
> 검수 단어: `24 Preludes`  
> 테스트 문장: `집중할 때 듣기 좋은 24 Preludes를 추천해드릴게요.`  
> 확인 내용: 제목에 포함된 숫자, 영문 복수형, 목적격 조사와의 연결

> **문장 제작 — 공개용 대체 예시 3**
>
> 검수 단어: `BWV 846`  
> 테스트 문장: `바흐의 BWV 846을 차분한 아침 음악으로 골랐어요.`  
> 확인 내용: 알파벳 약어, 작품목록 번호, 조사 앞 끊어읽기

테스트 문장은 발음을 유도하기 위한 기호의 나열이 아니라 실제 서비스에서 사용할 법한 안내 문장으로 구성했습니다. 단어가 문장의 앞, 가운데, 끝에 들어가는 경우를 모두 생각할 수 있었지만, 최소한 한 문장 안에서는 반드시 들어보는 것을 기준으로 삼았습니다.

테스트 문장은 수동 조사를 위한 도구라기보다 발음사전에 등록한 결과를 확인하는 도구였습니다. 주 단위로 등록 목록이 TTS에 반영되면 해당 단어가 포함된 문장을 재생해 발음과 호흡을 확인했습니다. 발음사전이 최종 서비스 문장에서 제대로 작동하는지 보기 위한 절차였습니다.

## 주 단위 등록 및 확인

발음사전은 한 번에 전체 항목을 등록하는 방식이 아니었습니다. 한 주 동안 조사하고 정리한 작업 목록을 매주 한 차례 TTS에 업데이트했습니다. 업데이트가 끝난 뒤에는 등록된 단어가 포함된 테스트 문장을 재생해 발음사전이 의도한 방식으로 적용됐는지 확인했습니다.

```text
주간 작업 목록 정리
  ↓
발음사전 등록
  ↓
주 1회 TTS 업데이트
  ↓
테스트 문장 재생
  ↓
Pass/Fail 확인
  ↓
필요 항목 재검수
```

이와 별도로 음악 서비스에는 매주 신규 곡과 아티스트가 추가됐습니다. 기존 약 3만 5천 개의 등록 후보를 처리하는 동안에도 새로운 발음 검수 대상이 계속 생겼습니다. 주간 작업에는 기존 후보뿐 아니라 새로 등록된 곡명과 아티스트명의 발음을 확인하고 사전에 추가하는 업무도 포함됐습니다.

결국 작업 대상은 고정된 목록이 아니었습니다. 기존 목록의 처리 속도와 신규 항목의 유입을 함께 고려해야 했습니다. 수동 조사만으로 전체 후보를 갱신하기 어려웠던 이유도 단순히 초기 항목 수가 많았기 때문만은 아니었습니다. 새로운 고유명사가 매주 추가됐기 때문입니다.

Pass/Fail은 발음을 찾는 단계에서 정답을 고르는 용도가 아니라, 사전에 등록한 발음이 실제 TTS 문장에 적용됐는지 확인하는 용도로 사용했습니다. 기대한 발음과 호흡으로 출력되면 Pass, 등록 내용이 반영되지 않았거나 명백한 오독이 남아 있으면 Fail로 구분했습니다.

## 발음과 끊어읽기의 구분

주 단위 테스트 문장을 듣다 보니, 어색한 출력이 모두 발음사전의 문제는 아니라는 점도 확인했습니다. 발음 오류와 끊어읽기 문제는 함께 들릴 수 있지만 수정 방향이 달랐기 때문에 두 유형을 구분했습니다.

발음 오류는 특정 표기를 어떤 소리로 읽는가의 문제였습니다. 아티스트명이나 곡명을 국내에서 통용되는 방식과 다르게 읽거나, 숫자와 약어를 잘못 읽는 경우입니다. 같은 표기에서 오독이 반복된다면 발음사전 후보로 정리할 수 있었습니다.

끊어읽기는 문장 안에서 어디서 쉬는가의 문제였습니다. 한 글자도 틀리지 않아도 작품명 중간에서 쉬거나, Op. 표기와 곡 번호를 붙여 읽으면 사용자가 구조를 파악하기 어려웠습니다.

> **끊어읽기 — 공개용 재구성 예시**
>
> 문장: `다음 곡은 베토벤의 교향곡 5번 다단조 작품 67입니다.`
>
> 문제 방향: `다음 곡은 베토벤의 교향곡 / 5번 다단조 작품 / 67입니다.`  
> 기대 방향: `다음 곡은 / 베토벤의 교향곡 5번 다단조 / 작품 67입니다.`

저는 끊어읽기를 쉽게 설명할 때 ‘AI가 자연스럽게 호흡해야 하는 위치’를 찾는 일이라고 이해했습니다. 문장을 먼저 읽고 의미 단위를 나눈 뒤, TTS의 실제 호흡과 비교했습니다. 고유명사 앞뒤에서 불필요하게 끊기지 않는지, 한 호흡이 지나치게 길지 않은지, 조사 앞에서 이름이 잘리지 않는지를 확인했습니다.

검수 과정은 다음 순서로 진행했습니다.

```text
검수 대상 확인
  ↓
테스트 문장 제작
  ↓
TTS 전체 문장 청취
  ↓
발음과 끊어읽기 구분
  ↓
Pass/Fail 기록
  ↓
기대 발음 또는 수정 방향 정리
  ↓
반영 후 재검수
```

문제가 들렸을 때는 해당 단어만 다시 듣지 않고 문장 전체를 확인했습니다. 앞 문장의 끝과 다음 문장의 시작, 조사와의 연결, 문장 길이에 따라 호흡이 달라질 수 있었기 때문입니다. 발음사전에 반영된 뒤에도 같은 문장으로 다시 들어야 실제 출력이 자연스러운지 확인할 수 있었습니다.

## 클래식 곡명에서 나타난 반복 문제

검수 범위에서 특히 규칙화가 필요했던 영역은 클래식 곡명이었습니다. 대중음악의 곡명은 고유한 제목이 중심인 경우가 많지만, 클래식 곡명에는 작품의 종류, 조성, Op. 표기, 곡 번호, 악장이 일정한 구조로 반복됩니다.

예를 들어 다음과 같은 표기가 나타날 수 있습니다.

```text
작품명 + 조성 + Op. 표기 + 곡 번호 + 악장
```

각 요소를 따로 보면 익숙한 표기지만, 하나의 곡명 안에 들어가면 숫자가 여러 번 등장합니다. TTS가 숫자의 역할을 구분하지 못하면 Op.와 곡 번호를 같은 방식으로 읽거나, 악장 번호를 제목의 일부처럼 이어 읽을 수 있었습니다.

### Op. 표기와 곡 번호

> **공개용 대체 예시**
>
> 표기: `Nocturne in E-flat Major, Op. 9 No. 2`  
> 발음 방향: `녹턴 이 번 내림마장조, 작품 구의 이`  
> 확인 내용: `Op.`와 `No.`의 역할, 두 숫자의 관계, 조성 뒤 호흡

### 교향곡 번호와 조성

> **공개용 대체 예시**
>
> 표기: `Symphony No. 5 in C Minor, Op. 67`  
> 발음 방향: `교향곡 오 번 다단조, 작품 육십칠`  
> 확인 내용: 교향곡 번호, 조성, Op. 표기의 구분

### 작품목록 번호

> **공개용 대체 예시**
>
> 표기: `The Well-Tempered Clavier, BWV 846`  
> 발음 방향: `평균율 클라비어곡집, 비더블유브이 팔백사십육`  
> 확인 내용: 작품목록 약어와 번호를 분리해 읽는 방식

### 수량이 포함된 작품명

> **공개용 대체 예시**
>
> 표기: `24 Preludes, Op. 28`  
> 발음 방향: `이십사 개의 전주곡, 작품 이십팔`  
> 확인 내용: 제목에 포함된 수량과 Op. 표기의 구분

<small>※ 위 발음은 내부 사전 원문이 아니라 작업 방식을 설명하기 위한 대체 예시입니다. 공개 가능한 규칙이 확인되는 경우에만 실제 사례로 교체합니다.</small>

클래식 곡명을 검수하면서 개별 오류를 하나씩 수정하는 방식만으로는 같은 구조가 반복될 수 있다고 보았습니다. 같은 표기 규칙이 여러 곡명에 나타난다면 공통 기준으로 묶고, 예외가 있는 항목만 개별적으로 다루는 편이 작업을 정리하기에 적절했습니다.

## 내부 읽기 기준 보완

등록 목적을 조정한 뒤에는 수동 기록에서 확인한 패턴을 바탕으로 내부 읽기 기준을 보완했습니다. 모든 고유명사의 정답을 하나의 규칙으로 만드는 것이 아니라, 반복되는 구조에서 명백한 오독을 줄일 수 있는 범위를 정하는 작업이었습니다.

클래식 곡명의 Op., 곡 번호, 조성, 악장처럼 표기 구조가 일정한 항목은 규칙 기반으로 정리할 수 있었습니다. 숫자를 알파벳이나 특정 소리 대신 사용하는 경우에는 장르와 이름의 구성 방식을 함께 봐야 했습니다. 아티스트명과 고유한 곡명처럼 제작자의 의도가 크게 작용하는 항목은 자동 분류 결과만으로 확정하지 않고 수동 확인 대상으로 남겼습니다.

| 구분 | 판단 기준 | 처리 방향 |
|---|---|---|
| 반복 구조 | 같은 표기가 여러 곡명에 반복됨 | 내부 읽기 기준 적용 |
| 장르 관용 표현 | 숫자·알파벳이 장르 안에서 특정 방식으로 사용됨 | 장르 맥락을 참고해 후보 분류 |
| 고유명사 | 제작 의도와 통용 발음의 영향이 큼 | 수동 조사 유지 |
| 예외 항목 | 공통 기준 적용 시 명백한 오독이 발생함 | 예외로 별도 관리 |

<small>※ 구체적인 분류 기준·우선순위·예외 목록은 제외하고, 수동 기록에서 패턴을 찾아 자동 분류 후보와 수동 검토 대상을 나눈 과정만 설명했습니다.</small>

## Pass/Fail 기록

Pass와 Fail은 발음사전이 TTS에 업데이트된 뒤 테스트 문장으로 적용 여부를 확인할 때 사용했습니다. 정확한 발음을 확인할 수 있는 항목은 해당 발음과 호흡을 기준으로 봤습니다. 제작자의 의도까지 확인하기 어려운 항목은 사용자가 듣기에 명백히 틀리거나 이상한 발음인지를 우선 확인했습니다. 현재 기준에서 허용 가능한 발음이면 Pass, 등록 내용이 반영되지 않았거나 분명한 오독이 남아 있으면 Fail로 기록했습니다.

단순히 Fail만 표시하면 어느 부분을 수정해야 하는지 알기 어려웠습니다. 그래서 검수 단어, 테스트 문장, 실제 발음 방향, 기대 발음, 문제 유형을 함께 볼 수 있도록 정리했습니다.

당시 사용한 필드를 기준으로 기록 구조를 다시 만들면 다음과 같습니다.

| 검수 항목 | 테스트 문장 | 문제 유형 | 기대 방향 | 결과 |
|---|---|---|---|---|
| Symphony No. 5 | 다음 곡은 베토벤의 Symphony No. 5입니다. | 숫자 읽기 | 곡 번호를 작품명과 자연스럽게 연결 | Fail |
| BWV 846 | 바흐의 BWV 846을 추천해드릴게요. | 약어 발음 | 약어와 번호를 분리해 읽기 | Fail |
| 국내 통용 아티스트명 | ○○의 대표곡을 골랐어요. | 고유명사 | 국내에서 통용되는 발음 적용 | Fail |
| 한글 곡명 | 다음 곡은 ○○입니다. | 이상 없음 | 현재 발음 유지 | Pass |

<small>※ 실제 내부 시트 화면을 옮긴 것이 아니라, 당시 사용한 기록 필드와 공개용 대체 문장으로 구성한 표입니다.</small>

Pass 항목도 기록에서 제외하지 않았습니다. 현재 발음이 정상적으로 유지되는 기준 사례가 있어야 수정 전후를 비교하고, 공통 규칙이 다른 항목에 영향을 주지 않았는지 다시 확인할 수 있었기 때문입니다.

원인을 적을 때는 TTS 모델의 내부 동작을 추측하지 않았습니다. “모델이 숫자의 의미를 이해하지 못했다”고 단정하기보다 “Op.와 곡 번호가 같은 방식으로 읽혔다”처럼 실제로 들은 결과를 기록했습니다. 인턴으로서 확인할 수 없는 원인보다 재현 가능한 입력과 출력의 차이를 전달하는 데 집중했습니다.

## 반영 후 재검수

발음 후보나 규칙이 사전에 반영된 뒤에는 같은 테스트 문장으로 다시 확인했습니다. 단어 자체의 발음이 수정됐더라도 문장 안에서 조사와 붙거나 숫자 앞뒤의 호흡이 달라지면 여전히 어색할 수 있었기 때문입니다.

재검수에서는 두 가지를 함께 확인했습니다.

1. 기존 Fail 항목이 기대한 발음과 호흡으로 바뀌었는가
2. 같은 규칙이 적용된 다른 항목에 새로운 문제가 생기지 않았는가

개별 고유명사를 수정한 경우에는 해당 문장을 다시 확인하는 것이 중심이었습니다. 공통 규칙을 적용한 경우에는 같은 구조를 가진 다른 곡명도 함께 살펴볼 필요가 있었습니다. 한 항목의 Pass만으로 규칙 전체가 적절하다고 단정하기 어려웠기 때문입니다.

> **재검수 — 공개용 재구성 예시**
>
> 최초 결과: Op.와 곡 번호를 구분하지 않고 이어 읽음 — Fail  
> 수정 방향: Op. 표기와 곡 번호 사이의 읽기 규칙 분리  
> 재검수: 같은 문장으로 발음과 호흡 확인  
> 추가 확인: 동일한 표기 구조를 가진 다른 곡명에서 예외 발생 여부 확인

이 기준을 적용하면 발음사전에 항목을 추가하는 데서 끝나지 않고, 실제 서비스 문장에서 수정 결과를 다시 평가할 수 있습니다.

## 적용 결과 및 한계

정리한 음악 도메인 발음사전 일부는 대화형 음악 추천 기능의 TTS와 사내 TTS에 적용됐습니다. 숫자와 영어가 포함된 곡명·아티스트명 약 3만 5천 개를 전체 등록 후보로 두고, 초기 약 10%인 3,500개가량을 수동으로 조사한 시점에 처리 방향을 바꿨습니다. 등록한 단어는 최소 한 개 이상의 테스트 문장으로 실제 TTS 적용 여부를 확인했습니다. 수동 조사 기록에서는 국적·장르·표기 구조에 따른 패턴을 찾았고, 클래식 곡명의 반복 표기와 숫자·알파벳의 사용 방식을 내부 읽기 기준으로 보완하는 데 의견을 보탰습니다.

제가 직접 TTS 모델을 개발하거나 최종 발음 규칙을 결정한 것은 아닙니다. 발음을 수동으로 조사하고 조사 경로와 소요 시간을 기록했으며, 기록에서 발견한 패턴과 규칙 후보를 담당 멘토에게 공유했습니다. 주 1회 업데이트 뒤에는 테스트 문장으로 등록 여부를 확인했습니다. 사전 일부가 두 서비스에 적용된 사실까지는 확인했지만, 적용 전후의 오독률이나 사용자 만족도 변화는 확인하지 못했습니다.

처음에는 사람이 모든 항목의 정확한 발음을 직접 찾았습니다. 기록이 쌓인 뒤에는 반복 패턴을 분석하고 AI 기반 자동 분류를 시도했지만, 제작자의 의도가 중요한 이름에서는 수동 개입이 계속 필요했습니다. 정해진 기간 안에 전체 항목의 완전한 정답을 확인하기 어렵다는 점을 받아들이고, 명백한 오독을 우선 줄이는 방향으로 발음사전의 등록 목적을 조정했습니다.

이 작업에서 가장 크게 달라진 것은 처리 도구가 아니라 평가 기준이었습니다. 확인 가능한 정확한 발음은 유지하되, 확인 비용이 큰 항목은 사용자가 듣기에 부자연스럽지 않은 수준을 우선 확보하고, 제작 의도 확인이 필요한 예외는 수동 검토 대상으로 남겼습니다.

## 최종 편집 자료

이 항목은 게시용 본문이 아니라 웹사이트 제작 전까지 유지할 작업 메모입니다.

1. **검수 대상 규모**: 약 3만 5천 개 항목을 보여주는 숫자 중심 시각 자료
2. **작업 방식의 변화**: 수동 조사, 기록 분석, 자동 분류 시도, 목적 재설정의 4단계 흐름
3. **수동 조사 예시**: 검색과 공개 자료를 통해 발음 후보를 찾는 과정을 대체 화면으로 구성
4. **패턴 발견**: 국적·장르·표기 구조가 발음 후보 분류에 미치는 관계를 도식화
5. **단어와 문장 비교**: 같은 곡명을 단독으로 들을 때와 문장 안에서 들을 때의 차이
6. **발음·끊어읽기 비교**: 동일 문장의 문제 호흡과 기대 호흡 표시
7. **클래식 곡명 구조**: 작품명, 조성, Op. 표기, 곡 번호, 악장을 색으로 구분
8. **품질 기준 변화**: 정확한 발음 탐색에서 명백한 오독 우선 제거로 바뀐 판단 과정
9. **Pass/Fail 기록**: 실제 내부 시트 대신 같은 필드로 만든 대체 자료
10. **서비스 적용**: 사내 TTS와 대화형 음악 추천 기능 TTS로 이어지는 적용 범위

실제 곡명 목록, 내부 발음사전, 원본 음성, 시트 화면은 사용하지 않습니다. 공개 가능한 사례가 확인되지 않은 부분은 대체 곡명과 새로 제작한 음성으로 설명합니다.
