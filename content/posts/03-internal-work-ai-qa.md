# 사내 업무용 AI의 답변 정확성과 실행 가능성을 검수하다

## 사내 정보를 찾아주는 대화형 AI의 초기 QA에 짧게 참여해 직원별 질문과 후속 대화를 테스트했습니다. 출처에 맞는 답변인지뿐 아니라, 실제로 실행할 수 없는 주문 같은 요청을 가능한 것처럼 안내하는 문제를 어떻게 구분했는지 정리했습니다.

---

## 프로젝트 소개: 사내 정보를 대화로 찾는 AI

회사 안에는 업무 규정, 복지 제도, 비용 처리 기준, 각종 신청 방법처럼 직원이 반복해서 찾아보는 정보가 많습니다. 필요한 문서가 어디에 있는지 알고 정확한 제도명을 검색할 수 있다면 직접 찾을 수 있지만, 신규 입사자나 익숙하지 않은 업무를 맡은 직원에게는 검색어를 정하는 일부터 어려울 수 있습니다. 문서를 찾은 뒤에도 자신의 상황에 적용되는 조건을 다시 읽어야 합니다.

제가 QA에 참여한 사내 업무용 AI는 이런 정보를 자연어 대화로 찾게 하는 초기 단계의 자체 서비스였습니다. 직원이 “야간 촬영이 끝난 뒤 교통비를 지원받을 수 있나요?”처럼 평소 쓰는 문장으로 물으면, AI가 사내 자료에서 관련 내용을 찾아 답변하고 확인할 수 있는 출처를 함께 제공하는 방식이었습니다. 여기서 출처는 답변의 근거가 된 사내 문서나 규정을 사용자가 다시 확인할 수 있도록 보여주는 정보입니다.

일반 검색과 달리 대화형 인터페이스에서는 후속 질문도 이어집니다. 첫 답변을 본 직원이 “외부 촬영장이고 자정이 넘을 것 같아요”라고 조건을 추가하면, AI는 앞서 물었던 야간 교통비 맥락을 유지한 채 답해야 합니다. 따라서 QA에서는 한 문장의 정답 여부뿐 아니라, 이전 대화의 조건이 유지되는지와 답변이 실제 사내 자료에 맞는지도 함께 확인해야 했습니다.

서비스가 다루는 요청은 정보 안내에서 끝나지 않을 수도 있었습니다. 사용자는 “휴가 신청 방법을 알려줘”라고 물을 수도 있고 “내 휴가를 신청해줘”라고 요청할 수도 있습니다. 두 문장은 같은 제도를 말하지만, 후자는 실제 시스템에 접속해 사용자를 확인하고 업무를 처리해야 합니다. 이 글에서 말하는 `실행 가능성`은 AI가 답변에서 약속한 주문·신청·예약을 실제 시스템과 권한으로 완료할 수 있는지를 뜻합니다.

저는 2025년 9월부터 10월까지 서비스의 매우 초기 단계에서 QA에 제한적으로 참여했습니다. 다른 인턴 업무보다 참여 기간과 비중은 작았습니다. 따라서 서비스 전체의 평가 체계를 설계하거나 개선을 주도한 경험으로 표현하지 않고, 직원의 업무 상황과 질문을 만들고 대화형 응답을 테스트한 범위만 정리했습니다.

담당한 업무는 서로 다른 직원의 업무 상황을 페르소나로 만들고, 각 페르소나가 물을 법한 자연어 질문을 작성해 답변을 확인하는 일이었습니다. 여기서 페르소나는 실제 직원 정보가 아니라 직무, 근무 장소, 시간과 질문 목적을 묶은 테스트용 사용자 상황입니다. 사내 정보와 답변의 일치 여부를 중심으로 정확성을 살펴봤고, 답변에 출처가 표시된 경우에는 해당 자료와 답변을 함께 확인했습니다.

| 구분 | 참여 내용 |
|---|---|
| 페르소나 | 직원의 직무·업무 상황별 사용자 설정 |
| 질문 제작 | 사내 정보·규정·복지·업무 관련 자연어 질문 작성 |
| 답변 검수 | 질문 의도와 답변 내용의 일치 여부 확인 |
| 근거 확인 | 표시된 출처가 있는 경우 답변과 함께 확인 |
| 대화 테스트 | 후속 질문에서 이전 맥락이 유지되는지 확인 |
| 문제 발견 | 실행할 수 없는 요청을 계속 이어가는 응답 확인 |

처음에는 정확한 정보를 제공하는지가 가장 중요한 평가 대상이라고 생각했습니다. 내부 사용자는 답변을 실제 업무에 참고할 수 있으므로, 사내 규정과 다른 정보를 전달하거나 질문에 필요한 조건을 빠뜨리면 다시 원문을 찾아야 합니다. 정보를 빠르게 찾기 위한 AI가 오히려 확인 업무를 늘릴 수 있었습니다. 먼저 서로 다른 직원이 실제로 어떤 표현으로 질문할지를 넓혀 보기로 했습니다.

제가 확인한 흐름을 단순화하면 다음과 같습니다.

```text
직원의 자연어 질문
  ↓
질문 의도와 이전 대화 확인
  ↓
관련 사내 자료 탐색
  ↓
답변과 출처 제시
  ↓
직원의 후속 질문 또는 다음 행동
```

<small>※ 위 흐름은 실제 내부 구조도가 아니라, 당시 제가 질문과 답변을 검수한 순서를 설명하기 위한 공개용 도식입니다.</small>

페르소나를 구체화할 때는 직무명만 바꾸지 않고 업무 장소·근무 시간·필요한 정보를 함께 묶었습니다. 같은 제도도 어떤 상황에서 왜 찾는지까지 정해야 후속 질문과 기대 답변을 구체화할 수 있었습니다.

## 답변 정확성 확인

사내 정보 검색형 AI의 답변은 일반적인 지식만으로 판단하기 어려웠습니다. 같은 제도나 업무 절차도 회사 내부 기준에 따라 적용 대상과 처리 경로가 달라질 수 있기 때문입니다. 답변이 자연스럽더라도 사내 정보와 다르면 업무에 활용하기 어려웠습니다.

초기 QA에서는 페르소나별 질문에 대한 답변이 질문 의도와 맞는지, 명확히 잘못된 정보가 없는지를 중심으로 확인했습니다. 답변에 출처가 표시된 경우에는 출처가 있다는 사실과 답변 내용을 함께 살펴봤습니다. 참여 기간이 짧아 출처 오류를 별도 유형으로 분류하거나 평가 기준을 정립한 것은 아닙니다.

> **공개용 재구성 예시**
>
> **질문**  
> 야간 촬영 후 교통비 지원은 어디에서 신청하나요?
>
> **확인할 내용**  
> 제도의 일반적인 설명으로 끝나지 않고, 질문자가 찾는 신청 경로를 안내하는가  
> 답변에 표시된 출처가 있다면 함께 확인할 수 있는가

이렇게 정보 질문을 확인하던 중, 답변의 정확성과는 다른 문제가 나타났습니다. 질문이 정보 검색에서 실제 업무 실행으로 바뀌었을 때도 AI가 같은 방식으로 대화를 이어간 것입니다.

## 멈추지 않는 주문 대화

테스트 질문의 범위를 넓히자 정보 검색만으로 끝나지 않는 요청이 등장했습니다. 사내 카페의 음료를 대신 주문해 달라는 요청이었습니다.

주문을 완료하려면 사용자 정보와 결제 수단이 연결돼 있어야 하고, AI가 실제 주문 시스템을 호출할 수 있어야 합니다. 당시 AI는 해당 정보를 확인하거나 주문 시스템을 실행할 수 없었습니다. 그럼에도 대화는 멈추지 않았습니다.

> **사용자**  
> 사내 카페에서 아이스 아메리카노 한 잔 주문해줘.
>
> **AI**  
> 네, 아이스 아메리카노 주문을 도와드릴게요. 원하는 옵션을 알려주세요.
>
> **사용자**  
> 기본으로 해줘. 금액은 사내 계정에서 차감해줘.
>
> **AI**  
> 네, 사내 계정에서 차감해 주문을 진행하겠습니다.

<small>※ 실제 원문이나 내부 화면이 아닌 공개용 재구성 대화입니다. 당시 확인한 문제 유형은 유지하고 서비스명·문장·세부 조건은 바꾸었습니다.</small>

문장만 보면 자연스럽습니다. 사용자의 요청을 이해했고, 다음 단계도 안내했습니다. 그러나 실제로는 주문을 넣을 수 없었고 사내 계정에서 금액을 차감할 수도 없었습니다. 필요한 정보가 무엇인지 묻는 것만으로 해결되는 문제도 아니었습니다. 해당 시스템과 연결돼 있지 않았기 때문입니다.

이 대화를 처음 봤을 때 확인해야 할 문제는 답변의 문법이나 친절함이 아니었습니다. AI가 할 수 없는 일을 할 수 있는 것처럼 말하고 있다는 점이었습니다. 그렇다면 답변 내용의 정확성만 확인하는 것으로 충분할까요?

## 정보 문의와 실행 요청의 구분

사내 AI에 들어오는 질문이 모두 정보 검색으로 끝나지는 않았습니다. 사용자는 “휴가 신청 방법을 알려줘”라고 물을 수도 있지만, “내 휴가를 신청해줘”라고 요청할 수도 있습니다. 두 문장은 비슷해 보여도 AI가 해야 하는 일은 다릅니다.

직원 유형만으로 기대 답변을 정할 수는 없었습니다. 같은 직원이 같은 휴가 제도를 묻더라도, 첫 문장은 정보를 확인하려는 질문이고 두 번째 문장은 실제 처리를 맡기려는 요청입니다. 질문에 등장한 단어보다 사용자가 대화를 통해 끝내려는 일이 무엇인지 구분해야 했습니다.

첫 번째 질문은 정보 제공으로 처리할 수 있습니다. 사내 규정과 신청 경로를 찾아 안내하면 됩니다. 두 번째 요청을 실제로 처리하려면 사용자 인증, 잔여 휴가 확인, 날짜 입력, 결재선 확인, 휴가 시스템 연동, 처리 결과 확인이 필요할 수 있습니다.

```text
정보 문의
“휴가 신청은 어디에서 하나요?”
→ 규정과 신청 경로 안내

실행 요청
“다음 주 금요일 휴가 신청해줘.”
→ 본인 확인, 권한, 시스템 연동, 처리 결과 확인 필요
```

정보 제공과 업무 실행을 구분하지 않으면 AI는 신청 방법을 알고 있다는 이유만으로 신청까지 할 수 있는 것처럼 답할 수 있습니다. 사내 카페 주문 사례도 같은 구조였습니다. 메뉴나 주문 방법을 설명하는 것과 실제 주문을 넣고 금액을 차감하는 것은 다른 작업이었습니다.

이 문제를 확인한 뒤에는 답변이 정확한지만 보는 것으로 충분하지 않았습니다. 답변이 약속한 행동을 AI가 실제로 수행할 수 있는지도 함께 살펴봐야 했습니다. 특히 실행할 수 없다는 사실을 한 번 놓치면, 이후의 자연스러운 추가 질문이 오히려 문제를 키웠습니다.

## 실행 불가능한 요청의 반복

실행할 수 없는 요청에서 나타난 문제는 한 번의 잘못된 답변으로 끝나지 않았습니다. AI는 사용자를 돕기 위해 필요한 정보를 계속 물으며 대화를 이어가려는 모습을 보였습니다. 일반적인 상담에서는 자연스러운 행동이지만, 실행 수단이 없는 상황에서는 질문을 더 받아도 업무를 완료할 수 없었습니다.

카페 주문을 예로 들면 음료 종류, 크기, 추가 옵션을 모두 확인하더라도 주문 시스템과 결제 계정이 연결돼 있지 않으면 실제 처리는 불가능합니다. 이때 추가 질문은 문제 해결을 위한 단계가 아니라 완료할 수 없는 대화를 연장하는 단계가 됩니다.

같은 위험은 다른 업무에서도 발생할 수 있습니다.

> **휴가 신청 — 공개용 재구성 예시**
>
> 사용자: “다음 주 금요일에 연차 신청해줘.”  
> 문제 응답: 휴가 종류와 사유를 묻고 신청을 진행하겠다고 안내  
> 누락된 조건: 사용자 인증, 잔여 일수, 결재선, 휴가 시스템 연동

<small>※ 위 휴가 신청은 실제 내부 대화가 아니라, 사내 카페 주문에서 확인한 문제를 다른 업무 유형에 적용한 대체 사례입니다.</small>

## 실행 불가 응답의 위험

실행할 수 없는 요청을 계속 이어가는 문제는 단순히 답변이 장황하다는 데 그치지 않았습니다.

첫째, 사용자가 업무가 처리되고 있다고 오해할 수 있습니다. “주문을 진행하겠습니다” 또는 “휴가를 신청하겠습니다”라는 문장은 실제 실행을 약속하는 표현입니다. 사용자가 완료 여부를 다시 확인하지 않으면 주문, 신청, 예약이 누락될 수 있습니다.

둘째, 사용자의 시간이 낭비됩니다. AI가 여러 차례 추가 정보를 요청하면 사용자는 처리가 가능하다고 믿고 대화를 이어갑니다. 마지막에 실행할 수 없다는 사실을 알게 되면 같은 내용을 실제 시스템에 다시 입력해야 합니다.

셋째, 불필요한 연산과 비용이 발생합니다. 완료할 수 없는 작업을 두고 대화를 반복해도 서비스 목적에는 가까워지지 않습니다. 요청의 실행 가능성을 앞에서 판단하지 못하면 대화 길이와 처리 비용만 늘어날 수 있습니다.

넷째, 사내 AI에 대한 신뢰가 떨어질 수 있습니다. 업무용 AI는 정보 탐색 시간을 줄이기 위해 사용합니다. 실제로 처리하지 못한 일을 완료한 것처럼 표현하는 경험이 반복되면, 사용자는 정확한 정보 안내까지 다시 확인하려 할 수 있습니다.

이 문제에서 중요한 것은 AI가 사용자의 의도를 이해했는지가 아니었습니다. 주문 요청을 정확히 이해했더라도 실행 권한과 시스템 연동이 없다면 처리 가능한 업무가 아니었습니다.

## 대응 방식 검토

실행 불가능한 요청을 발견한 뒤에는 AI가 어느 시점에서 어떻게 멈추는 것이 적절한지 생각해 보았습니다. 모든 실행 요청을 바로 거절하면 실제로 안내할 수 있는 정보까지 제공하지 못할 수 있고, 반대로 계속 질문하면 사용자가 처리가 가능한 것으로 오해할 수 있었습니다.

### 추가 정보를 계속 확인하는 방식

사용자에게 메뉴, 날짜, 금액, 사유 등을 차례로 묻는 방식입니다. 시스템 연동이 있고 정보만 부족한 경우에는 필요할 수 있습니다. 그러나 연동 자체가 없다면 정보를 모두 받아도 실행할 수 없으므로 대화만 길어집니다.

### 담당 시스템으로 연결하는 방식

AI가 직접 실행하지 않고 사용자가 처리할 수 있는 사내 시스템이나 메뉴를 안내하는 방식입니다. 정확한 경로를 알고 있다면 사용자의 다음 행동을 줄일 수 있습니다. 다만 연결 기능이 실제로 제공되지 않는데 “이동했습니다” 또는 “신청 화면을 열었습니다”라고 표현해서는 안 됩니다.

### 실행 불가만 안내하는 방식

처리할 수 없다는 사실을 즉시 알리는 방식입니다. 오해는 줄일 수 있지만, 사용자가 직접 처리할 방법까지 제공하지 않으면 도움이 부족한 답변이 될 수 있습니다.

### 가능한 범위와 불가능한 범위를 나누는 방식

AI가 직접 주문하거나 신청할 수 없다는 점을 먼저 알리고, 대신 관련 규정, 준비해야 할 정보, 처리 경로를 안내하는 방식입니다. 사용자가 완료 여부를 오해하지 않으면서도 다음 행동에 필요한 도움을 받을 수 있습니다.

네 가지 대응을 비교한 뒤에는 실행 불가 사실을 명확히 안내하면서 가능한 정보 제공을 이어가는 방식이 가장 적절하다고 정리했습니다. 중요한 것은 대화를 무조건 짧게 끝내는 것이 아니라, 실행을 약속하는 대화는 멈추고 정보 안내로 전환하는 것이었습니다.

## 실행 가능성 평가 기준

그렇다면 실행 요청을 다시 테스트할 때 무엇부터 확인해야 할까요? 답변 문장만 보기보다 해당 요청을 완료하는 데 필요한 조건을 먼저 살펴볼 수 있습니다. 저는 사내 카페 주문 사례를 바탕으로 다음과 같은 판단 기준을 정리해 보았습니다.

| 확인 항목 | 확인할 내용 |
|---|---|
| 시스템 연동 | AI가 실제 처리 시스템에 연결돼 있는가 |
| 사용자 인증 | 요청한 사람의 신원을 확인할 수 있는가 |
| 실행 권한 | 해당 사용자의 권한으로 업무를 처리할 수 있는가 |
| 필수 정보 | 주문·신청·예약에 필요한 정보가 갖춰졌는가 |
| 상태 확인 | 처리 성공 또는 실패 여부를 확인할 수 있는가 |

이 기준은 실제 프로젝트의 내부 정책이나 제가 확정한 평가 체계가 아닙니다. 당시 발견한 문제를 포트폴리오에서 설명하기 위해 사후에 정리한 평가 기준입니다.

다섯 가지 가운데 하나라도 충족되지 않았다고 무조건 같은 답변을 해야 하는 것은 아닙니다. 필수 정보만 부족하고 시스템 연동과 권한이 있다면 추가 질문을 할 수 있습니다. 반면 시스템 연동이나 실행 권한이 없다면 정보를 더 받아도 처리할 수 없으므로 실행을 약속하지 않아야 합니다.

```text
실행 요청 감지
  ↓
시스템 연동 여부 확인
  ├─ 연동 없음 → 실행 불가 안내 + 처리 경로 제공
  └─ 연동 있음
       ↓
사용자 인증·권한 확인
  ├─ 확인 불가 → 실행 중단 + 인증 방법 안내
  └─ 확인 가능
       ↓
필수 정보 확인
  ├─ 정보 부족 → 필요한 정보만 추가 질문
  └─ 정보 충족 → 실행 후 결과 상태 확인
```

이 구조를 적용하면 AI가 언제 질문을 이어가고 언제 멈춰야 하는지 구분할 수 있습니다. 자연스러운 대화를 오래 이어가는 것보다, 현재 시스템에서 가능한 범위를 정확하게 말하는 것이 우선이었습니다.

## 응답 개선 방향

실행할 수 없는 요청에 대한 답변은 네 부분으로 나누어 볼 수 있습니다.

1. 현재 AI가 직접 실행할 수 없는 업무임을 먼저 안내
2. 실행에 필요한 연동이나 권한이 없다는 이유를 간단히 설명
3. 사용자가 직접 처리할 수 있는 경로 안내
4. AI가 대신 제공할 수 있는 정보 제시

사내 카페 주문 사례에 적용하면 다음과 같습니다.

> **문제 응답 — 공개용 재구성**
>
> “네, 아이스 아메리카노 주문을 진행하겠습니다. 원하는 옵션을 알려주세요.”

> **개선 방향 — 공개용 재구성**
>
> “현재 이 대화에서는 사내 카페 주문과 결제를 직접 처리할 수 없습니다. 주문은 사내 카페 주문 화면에서 진행해 주세요. 필요한 경우 메뉴와 주문 방법은 안내해 드릴 수 있습니다.”

개선 방향에서는 처리하지 못하는 일을 먼저 분명히 알립니다. 동시에 대화를 완전히 종료하지 않고, AI가 실제로 제공할 수 있는 도움을 제안합니다. 사용자는 주문 완료 여부를 오해하지 않으면서 다음 행동을 선택할 수 있습니다.

휴가 신청에도 같은 구조를 적용할 수 있습니다.

> **문제 응답 — 공개용 재구성**
>
> “다음 주 금요일 연차 신청을 진행하겠습니다. 휴가 사유를 알려주세요.”

> **개선 방향 — 공개용 재구성**
>
> “현재 이 대화에서는 휴가 신청을 직접 등록할 수 없습니다. 휴가 신청은 사내 근태 시스템에서 진행해야 합니다. 신청 경로와 필요한 정보는 안내해 드릴 수 있습니다.”

“등록했습니다”, “처리 중입니다”, “진행하겠습니다”와 같은 완료·실행 표현은 실제 상태를 확인할 수 있을 때 사용하는 것이 적절하다고 보았습니다. 실행 결과를 확인할 수 없다면 사용자가 직접 처리해야 하는 정보 안내와 분리할 필요가 있었습니다.

## 사후 평가 기록안

당시에는 참여 기간이 짧아 별도의 평가 기록 체계를 만들지 않았습니다. 포트폴리오를 준비하며 같은 문제를 다시 검토할 수 있는 형태로 정리하면, 입력과 응답, 실제 실행 조건, 사용자에게 생길 수 있는 오해를 나눌 수 있습니다.

당시 경험을 바탕으로 사후 평가 기록을 구성하면 다음과 같습니다.

| 기록 항목 | 공개용 재구성 내용 |
|---|---|
| 사용자 요청 | 사내 카페 음료 주문 및 계정 차감 요청 |
| AI 응답 | 옵션을 추가로 묻고 주문을 진행하겠다고 안내 |
| 실제 가능 범위 | 카페 주문·결제 시스템 미연동 |
| 문제 | 실행할 수 없는 업무를 처리 가능한 것처럼 표현 |
| 사용자 위험 | 주문 완료 오인, 시간 낭비, 재입력 발생 |
| 기대 방향 | 실행 불가 안내 후 주문 경로 또는 메뉴 정보 제공 |

<small>※ 실제 내부 문서가 아니라, 당시 확인한 문제를 평가 항목으로 다시 구성한 자료입니다.</small>

원인을 정리할 때도 확인되지 않은 내부 동작을 단정하지 않았습니다. “모델이 사용자를 붙잡기 위해 거짓말했다”고 쓰기보다 “시스템 연동이 없는 요청에서 실행을 약속하며 추가 질문을 이어갔다”고 적으면 관찰한 사실과 평가자의 해석을 구분할 수 있습니다.

문제 유형은 다음과 같이 정리할 수 있었습니다.

- 정보 문의와 실행 요청의 구분 실패
- 시스템 연동 여부 확인 없이 처리 약속
- 인증 및 권한 확인 누락
- 완료 상태를 확인할 수 없는 응답
- 실행 불가능한 상황에서 불필요한 추가 질문 반복
- 사용자가 직접 처리할 수 있는 경로 안내 부족

이 분류는 당시 프로젝트에서 사용한 공식 체계가 아니라, 참여 경험을 바탕으로 포트폴리오에서 평가 관점을 설명하기 위해 재구성한 것입니다.

## 참여 결과 및 한계

이 프로젝트는 짧게 참여한 QA 업무였고, 제가 실행 가능성 평가 체계를 직접 적용하거나 서비스 정책을 결정한 것은 아닙니다. 답변 정확성과 출처를 확인하는 과정에 참여했고, 실행할 수 없는 요청에서도 AI가 처리를 약속하며 대화를 이어가는 문제를 발견했습니다.

따라서 이 글에서 말할 수 있는 결과는 문제를 발견하고 검수 관점을 정리한 범위까지입니다. 해당 문제가 최종 서비스에서 어떤 방식으로 수정됐는지, 수정 전후 지표가 얼마나 달라졌는지는 확인하지 못했습니다.

다만 이 경험을 통해 정보 검색형 AI의 답변은 세 층위로 나누어 살펴볼 수 있었습니다.

```text
답변 내용이 맞는가
  ↓
표시된 출처가 답변을 뒷받침하는가
  ↓
AI가 약속한 행동을 실제로 실행할 수 있는가
```

자연스러운 답변이 업무에 도움이 되려면 직원이 정보를 찾는 중인지, 실제 처리를 요청하는지, 처리 결과를 확인할 수 있는지를 함께 봐야 했습니다. 대화를 계속하는 것보다 사용자가 잘못된 완료 상태를 믿지 않도록 가능한 범위를 먼저 밝히는 편이 중요했습니다.

## 최종 편집 자료

이 항목은 게시용 본문이 아니라 웹사이트 제작 전까지 유지할 작업 메모입니다.

1. **도입 대화**: 사내 카페 주문 실패 사례를 메신저형 대화 화면으로 제작
2. **정보 문의와 실행 요청 비교**: 휴가 문의·신청 문장을 나란히 배치
3. **출처 검수 예시**: 질문, 답변, 출처, 누락 조건을 한 화면에 표시
4. **실행 가능성 판단 흐름**: 연동·인증·권한·필수 정보·상태 확인 순서도
5. **응답 전후 비교**: 카페 주문, 휴가 신청 사례를 인용 블록으로 강조
6. **위험 정리 자료**: 완료 오인, 시간 낭비, 비용 증가, 신뢰 저하를 간단한 도식으로 표현
7. **QA 기록 예시**: 실제 내부 화면 대신 같은 필드로 제작한 대체 기록표

내부 서비스명, 실제 화면, 사내 규정 원문, 사용자 정보, 시스템 주소는 사용하지 않습니다. 실제 경험에서 확인한 문제 유형은 유지하되, 예시 문장과 화면은 공개용 자료로 새로 제작합니다.
